Education


This section contains summary of what I did and what I learned in some of my favorite classes. My education also includes several math classes, physics, and some electrical engineering classes, but my favorite classes involve low level programming and microprocessors.

Senior Design

For my senior design project, I worked with a group of 8 other students to design and implement a sensory wall for a near by elementary school.
The first thing the team decided on was what we were going to create. We decide upon making 4 games which we decided on as a group. The 4 games we made were Bop it, Whack-a-mole/Simon-Says, a maze wheel and an LED matrix drawing game. During the initial design phase of the project, one of our main concerns was how we were going to mount the games to the wall. As a group we decided that back plate for each game would be consistent across all modules except for the maze wheel. We found that toggle bolts would be a good solution to mount our back plates to the wall. There were many other small details that we discussed as a group such as power routing and finding the specific hardware that we would need.
We split into pairs to implement each game. I worked on the LED matrix art game that we named "Picture This". My partner and I spent a lot of time figuring out how we were going to control the game. As opposed to the other games, we had much more creative freedom because this was not already a well defined game. After many meetings, we decided on a joystick to control the cursor, and several arcade push buttons to perform operations such as changing color, change draw mode, reset and erase. The art would be displayed on 2 64x32 matrices chained together to make one large 64x64 display. The microcontroller that we chose was a teensy 4. The main reasoning behind this was that there was a product called a "LED Matrix Smart Shield" which made driving the matrices much easier because we would not have to worry as much about the pin out that was required.
During our implementation phase, the main set back was waiting for parts. The main element that we were missing for a long time was the smart shield. While there was documentation that showed the pinout of the matrix, there was still some ambiguity that made testing our code difficult. One of the unfortunate side effects of using the smart shield was the library that we had intended on using was intended to be used in the Arduino ide. Due to this, the code isn't set up in a way that I would usually try to implement something like this, but this important thing is that everything worked in the end.
One cool thing the professor did for this class was allowing the class to vote on who had the best designed project (1st semester) and best implemented project (2nd semester) and whoever won the vote would get a guaranteed A for that semester. My group happened to win the vote for the best implementation.
The code for these project can be found on Github.

Control Systems

This was another project based class and the goal was, by the end of the semester to have a working control system to be able to show at the engineering demo day.
I worked in a group of 4 and we decide on making an automated blinds control system. Our inputs for the system were temperature and light level. The amount that the blinds were opened adjusted according to those two inputs using 2 stepper motors.
The first thing we brainstormed on was how we were going to design the blinds - would they be vertical or horizontal, could they be opened manually. We decided on vertical blinds and had originally planned to only use 1 stepper motor, but after testing we found that the wire that held the blinds would pinch and cause the blinds on the end to not be able to behave properly. Switching to two steppers, one on each end solved this problem. Something that we were not able to solve in the timeframe that was given was there were times when the temperature sensor would read corrupt values and cause the system to get stuck in its current state until a correct reading was read. As it turns out, temperature is difficult for cheap sensors to read. It may have been better to find a better sensor that could provide a more consistent reading.
In order to have a nice prototype for demo day, we built a small window. I designed and 3D printed some mounts to hold the stepper motors in place. We also 3D printed the blinds to give them a more consistent look as opposed to cutting them from some other material. We also found some spare acrylic to act as the "glass" for the window.
The code for these project can be found on Github.

Operating Systems

This class is less "flashy" to talk about, but I learned a lot about operating systems and how the kernel works. The projects for this class were based on Pintos, which is an incomplete operating system and through out the semester we implemented threads, scheduling, user programs, and virtual memory.
For the threads project, we used synchronization primitives' such as semaphores and locks to protect the threads critical section. We also implemented 2 forms of scheduling which included MLFQS and priority.
The next part of the project was to implement a way for user programs to be run on the OS. I enjoyed this part much more than the first because it was more memory management oriented. This involved stack packing process call inputs, and working with the file system. We also had to keep track of what was considered user memory vs kernel memory and if a program tried to access invalid memory a page fault would be sent.
We also implemented virtual memory which was much more on the theoretical side of things. We learned that virtual memory allows the OS to act as if it has infinite memory by keeping the active processes in RAM and inactive processes on disk. When a context switch is necessary the OS will determine if it needs to move data out of RAM onto disk if there is not enough space for the new process or just place the new processes data into RAM.

Embedded Systems

For this class a Nucleo-144 was used. I implemented 2 major projects, a timer system and what I called an "Alternative Hearing Device".
The timer project used a 4x4 keypad to control the timer and a 16x2 LCD to display the time and prompts. A string of number inputs were input and parsed to be converted into minutes and seconds so that the time could be decremented via a timer interrupt every second. I wrote the code with the intention of allowing the time to go beyond 9:59, but due to the restrictions of the project description it is currently set to not go past the ten minute mark.
The "Alternative Hearing Device" used a series of sound transducers to collect sound at different levels and displayed the sound level on the LCD as well as provided physical feedback via a vibration motor. A rotary encoder was used to adjust the intensity of the vibration motor. There are things for this project that I would have liked to do differently, but I was restricted to the scope of the class.
The code for these project can be found on Github.

Microprocessors

This course has been my favorite class that I have taken so far. This course utilizes the Tiva C processor model EK-TM4C123GPM which uses an ARM Cortex-M4 Processor. All the programming in this class has been done in ARM assembly. I have learned how to utilize GPIO from the microprocessor at a 'bare-metal' level by using bitwise control. Interrupts via GPIO buttons and the clock were used and initialized using bitwise control.

I implemented a version of the game "FlowFree". This project is using everything that I have learned throughout the class. I am used ANSI escape sequences to provide color to the game and several custom made data structures in order to track positions of the links and cursor movement. One of my favorite aspects of programming in assembly is pointer math (which I also enjoyed when programming in C). I find being able to manipulate raw memory very rewarding. The repository can be seen here (asm is in lab6 folder): FlowFree

Computational Intelligence

In this class, I learned about how computational intelligence techniques can be used in the context of hardware. There were 4 major projects for this class which included: Fuzzy Controller, Time Series, Microgrid Controller, and a written paper to summarize a IEEE technical paper. All the coding projects were completed in MATLAB.

The time series project was an implementation of representation techniques over time and how to classify classes into specific groups with a reduction in the number of dimensions used. The representation techniques used were PAA and SAX and the classification types were Euclidean and Manhattan. Project on Github: Time Series

The project that I enjoyed the most in this class was the Microgrid project. This project used a Genetic Algorithm to predict the best power output from 4 varying sources. The GA used 2 survival techniques: elitism and random selection as well as 2 reproduction techniques: mutation and crossover. More information on the implementation can be seen here: Implementation.
The project on Github: Microgrid

Systems Programming

This class had many labs and projects that were programmed in C. Prior to this class, programming was not coming easy to me, but this is where it all started to click for me. In this class, I learned memory manipulation, and pointer math. These skills have vastly helped me program in assembly, because memory manipulation is so important. Unfortunately I am unable to share some of the code that was produced as a part of this class, due to UB's strict academic violation policy, but I can share what was done.

One of the major projects that was included in this class was one where I had to develop an implementation of the C functions: malloc(), calloc(), realloc(), and free(). In order to create this implementation, sbrk() was used in order to create a chunk of memory that was accessible. I then used a linked list that stored valid memory addresses that were 4 bytes apart. malloc, pulled the required amount of memory off the linked list, and free put memory back onto the linked list. While this, is an over simplified version of what complexities were involved in this program, this project boosted my confidence significantly as a developer.

Calendar

This project started as a simple new tab page. I didn't like the way that chromes dark mode new tab looked, so I thought that I could do it better. This started out by having the search bar and the icons on the top right of the screen and it evolved from there.

I began this project a couple weeks before Covid 19 caused everything to go online. I had the idea to incorporate my calendar into this new tab shortly before this had happened. During this time I had been in a data structures class, so it was the perfect time to take on this type of project and it is also the reason I chose to use Scala. I knew there were ways that I could download the ical file automatically, it was just a matter of parsing the file. This code has undergone several ground up rebuilds and at this moment I would say it is perfect for my use case.

One of the main parts of this calendar were that I wanted to have the location displayed on the new tab page. After this implementation and after moving to online, I realized that I could place my class meeting links in the location field. After some regex to place the main site location as the name to be display, I know had a way to enter my lectures without having to search for a link for a specific class. When the semester ended I had the idea to make a full blown calendar using the same parsing code that I used for the new tab. Now, making a simple calendar that was placed in a simple html grid would have been easy, but I wanted the ability to have user input. I developed a way that I could click on a specific day and that days events would appear on the left of the calendar. I also included time indicators on each event. When an event is approaching it turns cyan and when it is current it turns red. There are a few other options that can manipulate this feature, like it flashes red if exam is in the description and it is currently an active event.

After the calendar was designed I purchased a raspberry pi that now sits in my room and displays my calendar all day with the help of some bash files to initiate parsing every thirty minutes and to power off the display at night.

There was a lot of thought that went into the design and implementation of this project. You can see the code for this project here: Calendar

x86 Assembly

I took some time to learn the basics of x86 assembly. The approach I took was to use the nasm assembler. This approach gave me access to the OS syscalls as apposed to my project that was done in my Microprocessors course which had no OS.

Before I was able to start this project I set up an Ubuntu Server VM with Virtual Box which I run in headless mode. This allows me to ssh into the VM with the VScode extension Remote-SSH. I am able to write all my code and run it with the built in terminal as if I was wasn't working off of a VM.

I started to learn the syntax by writing some useful subroutines that can be used for debugging such as: print_string, int_to_string, print_hex (which can print the value in a register) and a memory dump. The memory dump routine is particularly useful because it shows all the current values in memory given a starting address and size and it prints out as if I was using an IDE. This code can be found at ASM Library

After I built these debugging tools, I wanted to try to implement the game of life. I chose this because it was something I was familiar with, but also requires a bit of memory management. That code can be found at ASM GOL
